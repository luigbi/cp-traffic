CSI_Server.ini

CSI_Server is used to schedule Counterpoint applications to run at specific scheduled times.  Windows service programs run 24/7 and are automatically re-started if the server is re-booted. CSI_Server is the communications gateway between various Counterpoint applications. It uses settings defined in CSI_Server.ini to determine what file path locations to use. Unlike scheduling Utilities to run at Start up or through a Windows Task Scheduler, running programs through CSI_Server will allow you to monitor their progress, and will restart should they be aborted for any reason. 


CSI_Server can only be set up on the server. Programs can be scheduled to run on a dedicated PC at Start Up or through Windows Task Scheduler. All Counterpoint components must be registered and the PSQL Work Group must be installed on the dedicated PC.

Administrator Credentials 

When setting up or making changes to CSI_Server, you should be logged in with full administrator credentials in windows services. 


To access csi_server log on credentials and change them to admin credentials: 

  1. Search for Services
  2. Locate CSI_Server Server in the list  
  3. Right click and select properties
  4. In the logon tab, select use this logon and enter the user name and password that has full admin credentials on the server 


The CSI_Server.ini contains settings to control which applications are to be run and at what time. To make any changes to the CSI_Server.ini, contact support@counterpoint.net. 


The INI is divided into two sections: the Main Settings Section, which contains general information that affects the entire file, such as the database/Traffic.ini location, and the different Tasks (Section Names) that will be scheduled to run in through the CSI_Server.ini. The second section is Scheduled Tasks. This is where you setup the parameters of each task that is to be run.


CSI_Server loads the .ini file the first time it starts. After that it will only reload it if it detects that it has been changed. CSI_Server checks for a change at the same WakeUpInterval defined below. If the .ini file changes, the entries are merged into its internal list.

Main Settings Section

Main Settings

The first section contains entries that are relevant to all applications and provides the Traffic.ini location to the CSI Server.ini. 

  • Example: Database = X:\CSI\Prod\Data

Section Names =

This is a comma delimited list of all the defined sections/scheduled tasks in this file. 

  • Example: SectionNames = GetPaid, Backup, Matrix, AutoImport

Wake Up Interval =

This value defines how often CSI_Server wakes up and checks to see if it is time to start an application. This value is entered in seconds.

  • Example: WakeUpInterval = 2

Database = 

This is the location of the database.

  • Example: x:\CSI\prod\Data

TestDatabase = 

This is the location of the Test database.

Example: x:\CSI\test\Data

SaveData = 

This is the location of the folder in which Counterpoint backups are saved.

  • Example: x:\CSI\prod\SaveData

ZeroBTR = 

This is the location of the database Zero_BTR Files.

Example: x:\CSI\prod\zero_btr

LogFile =

This entry defines where CSI_Server logs its actions / failures. If not specified the log file is created in the same folder where CSI_Server is located.

  • Example:

LogFile= X:\CSI\Prod\Data\Messages\CSI_Server_Log.txt

Automatically Updated Fields

The following five fields are automatically updated by the Traffic and/or Affiliate system when a scheduled task completes.

LastBackupDate=

Once a backup is completed, this field will be updated with its date and time.

  • Example: 05-18-2015 06:45PM

LastBackupFileName =

This is the name assigned to the backup file, where T+A (Traffic and Affiliate (if applicable)), XYZ (Client name taken from Traffic>Site), 051815 (Date, mmddyy), V70 (Counterpoint version number), and DDF041315A.Zip (The date of the DDF files, mmddyy).

  • Example: T+A_XYZ_051815_V70_DDF041315.ZIP

LastClearReportStatus=

This indicates the number of files reset/cleared and any errors that may have occurred. 

  • Example: Files reset = 18. Errors = 0

LastCopyDate=

This is the date the production database was last copied into Test.

LastBackupStatus=

This displays information about the last backup.

  • Example: Backup type selected is Traffic & Affiliate

CSI_Server Application Execution

Each application section contains the entries that define when the program is to run.

[Task Name/Section Name]

This is the name of the task. This must exactly match the name defined in Section Names.

Enabled


0 = disabled, 1 = enabled

WeekDays

Week days are Sun Mon Tue Wed Thr Fri Sat.  Example: WeekDays = 0111110

StartTime

The time of day to start the application. The time can be entered in several different formats. Example: 2pm, 14:00, 2. Seconds can also be defined. 2:10:45pm.

WeekNumber

This is used to specify which broadcast week to run in.  To specify that the task will only be run during the second broadcast week. Example: WeekNumber = 2

MonthsDaysToRun

This can be used to specify a specific day or days to run and will run the task only on these days.  Valid ranges are 1-31. Any values outside of this range will be considered as all days being specified. For instance, zero or blank effectively turns this setting off and the task will be run on all days. 

Example: MonthDaysToRun = 1, 15, 30

NOTE: When this setting is used, WeekDays would normally only have one day turned on. For example, if WeekNumber is set to 2, then WeekDays would be set as 0000001, which would run the task in the second broadcast week on Sunday only.

Interval

Defines the next time the application is to be run.   

Example:  If the interval is 3600 and the start time is 2P, the program will run at 2P and every hour after that.  If the interval is 86400 (24 hours) it will be run once a day at 2P.

TimeSlot

This is used to specify that a task is only to be run between these hours. If either of the times specified cannot be converted properly due to a format error, TimeSlot is ignored. Leaving this setting blank would be the same as entering 12am, 12pm (All times on).

Example: TimeSlot = 6am, 6pm 

Exe

Defines the path and file name of the application to start.  Example: Exe = X:\CSI\Prod\Exe\xxxx.exe

StartInFolder

Defines the start in path where the traffic.ini or affiliat.ini is stored.  Example: StartInFolder = X:\CSI\Prod\Data

CommandLine

Defines the command line parameters if needed, to be passed to the application being started.        Example: CommandLine = /Debug

KeepProgramRunning

Defines whether CSI_Server is to watch and verify the application continues to run. If the application stops it will be restarted. CSI_Server checks these applications at the WakeUpInterval.        Example: KeepProgramRunning = 1

Matrix

There are several ways in which to setup the Matrix.ini.

CSI_Server.ini example with one set of export parameters

[Matrix] 

Enabled = 1 

WeekDays = 1111111 

StartTime = 6:00AM 

Interval = 86400 

Exe = x:\csi\prod\exe\Exports.exe 

StartInFolder = x:\csi\prod\data 

CommandLine = Auto-Matrix Section-Matrix Std Net 

KeepProgramRunning = 0  


Note: the entry "CommandLine" passes information to the application program,  in this case it's Matrix. It passes Key parameters that the program looks for. If it finds the text "Auto-Matrix", then the program knows it's running automatically, and no questions should be displayed/asked. Then, the following text next to "Section-" tells the application program which words to looks for in exports.ini so it can also pick up the options that Matrix will be using (for example, is it to run Calendar or Std, it is to include NTR, etc).
